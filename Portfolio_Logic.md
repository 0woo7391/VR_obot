# 🏆 Project Contribution & Logic Design (Portfolio)

**프로젝트명:** ROS2 기반 VR 로봇 원격 제어 시스템 (Sim-to-Real Teleoperation)
**담당 역할:** System Architect & Core Logic Designer

본 문서는 프로젝트 진행 과정에서 코딩(Implementation) 이전에 선행된 **핵심 로직 설계, 문제 해결 철학, 그리고 시스템 아키텍처 구축**에 대한 본인의 주요 기여 내역을 정리한 포트폴리오용 명세서입니다.

---

## 1. 근본 원인 분석(Root Cause Analysis) 및 시스템 무결성 유지
* **상황:** 6축 매니퓰레이터의 MoveIt2(OMPL) 플래닝 과정에서 `Catastrophic failure (Timeout)` 에러가 지속적으로 발생함.
* **로직 설계 및 조치:** 
  - 단순히 에러를 피하기 위해 전역 플래너 알고리즘을 타협(RRTstar -> RRTConnect)하려는 방향성을 배제함.
  - "문제가 생겼다고 구조를 뜯어고칠 것이 아니라, 현상의 근본 원인부터 파악해야 한다"는 엔지니어링 철학을 바탕으로 분석을 주도함.
  - 알고리즘의 최적화 연산에 필요한 시간이 물리적으로 부족하다는 것을 짚어내어, 코드 수정 없이 RViz의 `Planning Time` 파라미터를 튜닝하여 시스템 뼈대를 훼손하지 않고 완벽한 플래닝을 성공시킴.

## 2. 단계적 통합 로직 (Iterative Development Strategy)
* **상황:** VR(Meta Quest 2) 기기와 로봇(Ubuntu)을 다이렉트로 연결할 경우, 통신 에러와 렌더링 에러의 원인을 특정하기 매우 어려움.
* **로직 설계 및 조치:**
  - **[PC GUI -> 로봇 -> VR]** 로 이어지는 3단계 상호작용 아키텍처 로직을 설계함.
  - 무거운 VR 환경을 세팅하기 전, Unity 화면에 PC 마우스로 클릭할 수 있는 2D UI(버튼)를 먼저 구현하여 데이터 송수신과 로봇의 움직임을 독립적으로 검증하는 중간 단계를 도입함.
  - 이를 통해 에러의 변수를 완벽히 통제하고 개발 속도와 안정성을 대폭 향상시킴.

## 3. Sim-to-Real 아키텍처 제어 로직 설계
* **상황:** 가상 환경(Simulation)에서 성공한 코드를 실제 하드웨어(Real Robot)에 이식할 때 발생하는 막대한 코드 수정 비용과 물리적 충돌 위험성.
* **로직 설계 및 조치:**
  - 로봇의 '두뇌(MoveIt2)'와 '근육(ros2_control)'을 완벽히 분리하는 아키텍처 로직을 기획함.
  - `joint_limits.yaml`에 실제 하드웨어 스펙과 가속도/속도 한계를 강제 주입하여, 가상 환경에서도 실제 물리 법칙이 동일하게 적용되도록 설계.
  - 결과적으로 실행 런처(`launch.py`)만 교체하면 코드 단 한 줄의 수정 없이 가상에서 현실로(Sim-to-Real) 유연하게 전환되는 시스템을 완성함.

## 4. 프로젝트 체계화 및 형상 관리 시스템 도입
* **상황:** 복잡한 ROS2 패키지와 Unity 연동 과정에서 히스토리 추적 및 다음 스텝의 불확실성 발생.
* **로직 설계 및 조치:**
  - 프로젝트 폴더 내에 3단계 문서 관리 체계를 직접 고안 및 도입함.
    1. `Current.md`: 현재까지의 트러블슈팅과 완료된 상태 트래킹
    2. `PLAN.md`: 잠재적 리스크 분석 및 우회 방안이 포함된 미래 액션 플랜
    3. `System.md`: 시스템 전체 데이터 흐름과 훼손되지 않아야 할 핵심 설계 원칙 명시
  - 이 체계를 통해 개발 파트너(AI/팀원)와의 협업 효율을 극대화하고, 프로젝트의 영속성을 확보함.
